## How stories and work items are recorded in the backlog

What the process for this is.
[]: # - How are stories and work items prioritized?
 - priority is mainly driven by the client's most pertinent goals -
 - items are ranked vertically in the "to do" column -
 - most critical items placed at the top -
 -  review and adjust priorities accordingly

[]: # - How are stories and work items estimated?
- utilizing the tshirt methodology -
- small = quick fix or minor documentation update (less than a day) -
- medium = standard feature or component (1-3 days) -
- large = complex integration or architectural work (requires segmentation of problem)
  
[]: # - How are stories and work items assigned to team members?
- sub teams within the tribe select from the priority stack -
- members assign themselves to the github issue

[]: # - How are stories and work items tracked?
- done within the github projects kanban board -
-  to do = prioritized backlog -
-  in progress = actively working on it -
-  in review = a pull request is open and awaiting review -
-   done = merged onto main branch after testing
  
[]: # - How are stories and work items reported on?
- continuous updates are provided through git commit messages and PR descriptions -
- at the end of each milestone, review the "done" column in comparison to Rich's deliverables
  
[]: # - How are stories and work items archived?
- issues are closed when merged into the main branch -
- items in the done column remain as a historical record
